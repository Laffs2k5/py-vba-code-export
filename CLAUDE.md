## Overview

A small Python CLI (`vba-export`) that extracts VBA source from OOXML macro-enabled Office files (`.xlsm`, `.docm`, `.pptm`) and writes each module to a separate `.bas`/`.cls`/`.frm` file. Legacy binary formats (`.xls`, `.doc`, `.ppt`) are not supported. Built on [oletools](https://github.com/decalage2/oletools)' `VBA_Parser`.

## Commands

All commands assume the venv is active: `source venv/bin/activate`

```bash
pip install -e ".[dev]"                                   # install with dev tools (pytest, black, flake8)
vba-export <file> [output_dir] [-v]                       # run; output_dir defaults to ./exported_vba/
pytest                                                    # run all tests
pytest tests/test_exporter.py::TestVBAExporter::test_init # run a single test
pytest --cov=src/vba_export --cov-report=term-missing     # coverage
black .                                                   # format (line-length 100)
flake8                                                    # lint
```

## Architecture

Two modules under `src/vba_export/`:

- **`exporter.py`** — `VBAExporter` does all the work. `extract_vba_modules()` returns `{module_name: source}`; `export_to_directory()` writes them out. Files are written UTF-8 with `errors="ignore"`, so undecodable bytes are silently dropped rather than raising.
- **`cli.py`** — Click wrapper (`main`, the `vba-export` entry point defined in `pyproject.toml`). Catches all exceptions and reports via `click.echo(..., err=True)`.

**Extension detection** (`VBAExporter._get_extension`) is the one non-obvious piece: it infers the output extension from the module *name*, not its actual VBA type. A name already ending in `.bas/.cls/.frm` is kept as-is; names starting with `thisworkbook` or `sheet` become `.cls`; everything else defaults to `.bas`. This means UserForms are **not** detected as `.frm` by name and fall through to `.bas` unless `extract_all_macros()` already supplied the `.frm` suffix — keep this in mind when touching naming logic.

## Testing gotcha

Tests in `tests/test_exporter.py` and `tests/test_cli.py` depend on a **hardcoded absolute sample-file path** (`/mnt/c/Users/peder/Downloads/Kopi av PrepareEU_detailed cost report_MCF.xlsm`) via the `sample_xlsm_path` fixture. Tests that use it will fail on any machine where that file is absent. When adding tests, either reuse this fixture or supply your own `.xlsm`; don't assume a portable sample exists in-repo.

## Security-audit workflow

This tool is used to vet Office macros before signing them. `.github/prompts/vba-security-review.prompt.md` is a detailed VBA security-audit guide (network calls, shell execution, COM objects, data exfiltration, severity scoring). Typical flow: export both an old and new version, `diff -rq` the two export dirs, and audit only the changed modules against that guide.

Note: `Attribute VB_Base` GUIDs in `.frm` modules are auto-regenerated by the VBA designer on recompile and are **not** security-relevant — a diff showing only that line means no code changed.
